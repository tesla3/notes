# CLI Tools & Context Efficiency for Coding Agents

Research on how CLI tools, usage patterns, and architectural decisions affect context window efficiency in LLM-based coding agents.

**Date:** 2026-03-04
**Sources:** Academic papers, practitioner blogs, agent source code analysis, Reddit/HN discussions

---

## The Core Problem

In a typical coding agent session, **99% of tokens are input tokens** accumulated in the trajectory — only 1% are generated by the LLM (OpenRouter data for Claude Sonnet). Every tool call result stays in the trajectory and gets re-sent with every subsequent call. A verbose `ls` output or a full file read snowballs through the entire session.

> "If the LLM opens a large file or runs a command that generates verbose output, the computational cost of these trajectory tokens snowballs for each of the following steps, resulting in a large waste."
> — AgentDiet paper (Xiao et al., Sep 2025)

This means **the most important optimization for a coding agent isn't faster tools — it's tools that produce less output while preserving signal.**

## What the Research Shows

### 1. Agentic Search Beats RAG

Claude Code's team **deliberately abandoned RAG** (including Voyage embeddings) in favor of agentic search with simple CLI tools like `grep` and `rg`. Agentic search consistently outperformed RAG on both benchmarks and subjective quality evaluations.

Why: An agent that uses `rg` to find exactly what it needs, reads only the relevant sections, and builds understanding incrementally produces **higher-quality, more targeted context** than dumping pre-embedded chunks into the window.

Source: "An Exploratory Study of Code Retrieval Techniques in Coding Agents" ([preprints.org 202510.0924](https://www.preprints.org/manuscript/202510.0924), Oct 2025). **Note:** Previously misattributed to arxiv:2508.11126, which is actually "AI Agentic Programming: A Survey."

### 2. Token Efficiency Varies 10x Between Approaches

An empirical comparison of 7 coding agents (Claude Code, Gemini CLI, Codex CLI, Cursor, Amp, Cline, Aider) found:

| Agent | Approach | Token Usage |
|-------|----------|-------------|
| **Aider** | Tree-sitter AST + PageRank graph | **8.5-13k tokens** (4-6% context utilization) |
| **Amp** | Sub-agent delegation | ~19k (2% of 968k context) |
| **Cline** | ripgrep + fzf + tree-sitter AST | ~35k |
| **Claude Code** | Agentic search (grep/rg) | High — predictable but verbose |
| **Gemini CLI** | Agentic search (parallel tool calls) | High — fastest but token-heavy |
| **Codex CLI** | Shell command orchestration | Highest |

**Key finding:** All approaches successfully completed retrieval tasks. The difference was entirely in context efficiency.

Source: "An Exploratory Study of Code Retrieval Techniques in Coding Agents" ([preprints.org 202510.0924](https://www.preprints.org/manuscript/202510.0924), Oct 2025); ContextBench (arxiv:2602.05892, Feb 2026). **Caveat:** ContextBench's headline finding is that "sophisticated agent scaffolding yields only marginal gains" in retrieval quality — the 10x cost variation doesn't translate to proportional quality gains.

### 3. AgentDiet: Trajectory Reduction Works

AgentDiet (Xiao et al., Sep 2025) identifies three types of waste in agent trajectories:

- **Useless:** Cache files in directory listings, verbose build output
- **Redundant:** Repeated content in file edits (old content echoed in new)
- **Expired:** Files inspected during search that are no longer relevant

A cheap reflection LLM (GPT-5 mini, 12x cheaper than the main agent) prunes old trajectory steps via sliding window. Results:

- **40-60% input token reduction**
- **21-36% final cost reduction**
- **No performance degradation** (-1% to +2% solve rate change)
- Actually *improved* performance with Gemini 2.5 Pro by preventing context-induced loops

**Counterintuitive finding:** Removing tokens *improved* performance in some cases. The model reasons better with less noise.

### 4. Simple Observation Masking Beats Summarization

A study comparing context management strategies (Lindenbauer et al., Aug 2025) found:

- **Observation masking** (sliding window, M=10, older outputs replaced with placeholders): Halves cost, maintains or improves solve rate
- **LLM summarization**: Similar quality but higher cost (summary calls = ~7.2% of total cost) and counter-intuitively *longer* trajectories because smoothed-over failure signals cause the agent to retry more

**Verdict:** Simple truncation/masking is Pareto-optimal for verbose tool outputs. LLM summarization is only worth it for loop detection.

### 5. Focused Context Outperforms Large Context

Multiple studies converge on this:

- CompLLM: 2x compression with **4x faster TTFT** that actually *surpasses* uncompressed performance
- Retrieve-then-solve: Improved Mistral from 35.5% to 66.7% by being selective
- All 18 frontier models tested show performance degradation with context length (Chroma study)
- Effective context is **50-65% of advertised** window size
- 30%+ accuracy drops for content in the middle of long contexts (lost-in-the-middle)

> "A 200K model with clean, compacted context will outperform a 2M model drowning in noise tokens."

## The CLI Tools That Actually Matter

Based on what practitioners and agent developers are actually doing:

### Tier 1: Context-Efficient Search (the foundation)

**`rg` (ripgrep)** — The single most important tool for a coding agent.
- Respects `.gitignore` automatically → no `node_modules`/`venv` noise
- `-l` (files only), `-c` (counts only) → answer questions without dumping content
- `-t py` (file type filter) → scoped search
- Clean output format → directly parseable, no post-processing tokens wasted
- Used by: Claude Code, Gemini CLI, Cline, most agents as primary retrieval

> "Every search result an agent produces is dumped into a context window. Cleaner output doesn't just improve developer experience — it directly impacts how much work an AI agent has to do."

**`fd`** — Better `find` for the same reasons.
- Respects `.gitignore`, simple syntax, faster
- `-e py -d 2` → bounded, scoped file discovery

**`ast-grep` (`sg`)** — Structural code search at the AST level.
- Zero false positives for code patterns (unlike regex grep)
- r/ClaudeCode practitioners report putting `sg` as mandatory search tool in CLAUDE.md
- Cline uses tree-sitter AST as one of three retrieval tiers
- **Less noise = fewer tokens wasted on irrelevant matches**

### Tier 2: Structured Data Extraction

**`jq`** — Surgical JSON extraction.
- `jq '.keys'` or `jq '.data[0]'` → structure/sample instead of full dump
- Pipeline: `curl ... | jq '.results[] | .name'` → extract exactly what's needed
- Critical for API work where raw responses can be 10k+ tokens

**`sqlite3`** — SQL queries on structured data.
- Aggregate, filter, join before results hit context
- `SELECT count(*) FROM ...` → one number instead of thousands of rows

### Tier 3: Codebase Overview (high signal, low tokens)

**`tokei`** — Entire codebase structure in ~20 lines.
- Languages, LOC, file counts → instant project understanding
- Alternative to reading dozens of files to "get the lay of the land"

**`eza --tree -L 2`** — Bounded directory overview.
- Project structure without reading any files
- Depth-limited → predictable output size

**Aider's repo-map pattern** (Tree-sitter + PageRank):
- Parse AST → extract definitions → rank by cross-file references → fit to token budget
- Achieves 4-6% context utilization vs 54-70% for iterative search
- Available as standalone tool: [RepoMapper](https://github.com/pdavis68/RepoMapper)

### Tier 4: Output Discipline

**`| head -N`** — The most underrated context saver.
- Caps any exploratory command's output
- Should be default on anything with unknown output size

**`git diff --stat`** → what changed (file list) before `git diff` (full content)
**`git log --oneline -10`** → compact history
**`wc -l`** → file size before committing to read
**`2>/dev/null`** → suppress irrelevant stderr

## Architectural Patterns from Production Agents

### Sub-agents for Context Isolation

Claude Code, Amp, and others use **sub-agents** that get their own context window:

> "The subagent solution is to farm out the (X + Y) * N work to specialized agents, which only return the final Z token answers, keeping your main context clean."
> — Shrivu Shankar, "How I Use Every Claude Code Feature" (Nov 2025)

Claude Code's Explore sub-agent:
- Runs on Haiku (cheap/fast)
- Read-only: `Glob`, `Grep`, `Read`, safe bash (`ls`, `git status`, `find`, `cat`, `head`, `tail`)
- Returns summary to main agent → main context stays lean
- Main agent never sees the 50 files explored, only the 3-line answer

### CLI Over MCP for Token Efficiency

A practitioner who deleted 3 MCP servers in favor of direct CLI usage:

> "There is an ambient token cost to MCP: tool definitions must be persistently loaded into the agent's system prompt. A CLI call is just a bash command and its stdout — it only consumes tokens when actively used. For a simple file conversion, switching from MCP to CLI cut token usage by roughly 40%."

CLI tools are context-efficient because:
- No JSON-RPC framing overhead
- No persistent tool definition cost
- `--json` flag for structured output when needed
- `--quiet` for pipe-friendly bare output
- Agent already knows how to read `--help` and parse output

### Progressive Disclosure

Don't expose 100 tools upfront. Expose a discovery tool. The agent loads specific schemas only when needed. This applies to agent instructions too — Trail of Bits recommends:

> "Put stable context in CLAUDE.md, not the conversation. Scope work to one session. Prefer `/clear` over `/compact`."

## The Anti-Patterns

What wastes context, based on the research:

1. **Reading whole files when you need 10 lines** — Use `read` with offset/limit, or `rg -n` to find the exact location first
2. **`grep` instead of `rg`** — grep scans everything including node_modules, binaries; every junk match is a wasted token
3. **Full `git diff` when `git diff --stat` answers the question**
4. **Running commands without `| head` when output size is unknown**
5. **Dumping full API responses when `jq` can extract one field**
6. **Re-reading files that haven't changed** — expired context
7. **Verbose error output in trajectory** — only the actionable part matters
8. **LLM summarization as default compression** — simple masking is usually better and cheaper

---

## Verdict

The "power tools" question isn't about which flashy CLIs are installed. It's about **using any tool in the mode that minimizes output tokens while maximizing information**.

The hierarchy:
1. **Don't read it** — Can you answer without looking? (`rg -c`, `wc -l`, `git diff --stat`)
2. **Read the minimum** — Scope, filter, limit (`rg -l`, `jq '.key'`, `head -20`, `read` with offset)
3. **Read it structurally** — AST-level tools (`ast-grep`, `tokei`, repo-map)
4. **Read it fully** — Only when the above are insufficient

Every token in the context window is a cost — not just financially, but cognitively for the model. The research is clear: **focused context outperforms large context**. The best coding agents aren't the ones with the biggest windows; they're the ones that put the right tokens in front of the model.
